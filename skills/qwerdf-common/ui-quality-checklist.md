# UI Quality Checklist

本清单用于 UI 蓝图、Figma、前端实现和 UI Review。每一项都必须能用页面规格、Figma frame、截图、浏览器验证、测试或代码位置举证。

## 0. 低级视觉问题零容忍

以下问题不得作为“已通过”交付；发现后必须修复并用截图 / frame / 视口证据复查，或明确记录为待人工确认：

- 遮挡、遮掩、重叠：固定区、浮层、toast、drawer、popover、tooltip、文本、图标、badge、表格列或图表标签互相压住。
- 文本 / 容器溢出：长中文、英文、数字、金额、状态标签、按钮文案、面包屑、表格列撑破容器或被无意义裁切。
- 移动端不安全：侧向安全边距不足，页面级横向溢出，固定底部按钮 / toast / 导航遮挡提交区、错误提示或关键内容。
- 字体不一致：同语义层级混用字号、字重、行高或字体族；用手动缩小字体掩盖布局问题。
- 视觉系统漂移：圆角、阴影、边框、按钮高度、卡片 padding、表格密度、图标风格在同页面随机变化。
- 不整齐：tabs、筛选器、标题、列表内容、表单 label / input、按钮组、卡片和表格列明显错位或基线不齐。
- 状态不一致：default、loading、empty、error、permission、disabled、focus、hover、active、saving 状态缺失或视觉表达不统一。
- 结构性图标误用：用 emoji 当结构性图标，或混用无语义图标导致操作不可扫描。
- 产品类型错配：SaaS / CRM / 管理后台 / 操作台被设计成低密度营销页、装饰卡片堆砌或大面积 hero。

## 1. 视觉层级

- 页面主任务在首屏可识别，主按钮和危险按钮不混淆。
- 标题、筛选、内容、操作区和反馈区层级稳定；不要用同等强调表达不同重要性。
- 对管理后台和操作台，列表、表格、筛选和批量操作优先于装饰性 hero 或大图。

## 2. 布局与密度

- 页面宽度、栅格、侧栏、顶部栏、表格列和固定操作区有稳定尺寸。
- 左对齐、表单 label / input、按钮组、卡片网格、表格表头 / 内容和数字列必须成列；不得出现明显基线不齐或卡片高度随机。
- hover、loading、错误文案、badge、图标和按钮状态不会改变布局尺寸造成跳动。
- 文本不重叠、不遮挡前后内容；长文本、长单词、数字和状态标签有换行、截断或 tooltip 策略。
- 桌面和移动端无横向溢出；移动端关键操作不被固定栏遮挡。
- fixed header / footer / side nav、modal、drawer、popover、tooltip 和 toast 不遮挡主操作、表单字段、错误提示或关键数据。

## 3. 颜色、字体与图标

- 颜色使用 `ui/ui-design-system.md` 中的语义 token；成功、警告、错误、禁用、选中状态不能只靠颜色区分。
- 字号符合页面密度：紧凑面板、表格、侧栏不用 hero 级文字。
- 同层级字体族、字号、字重、行高保持一致；不得随机混用字体或在同一层级切换字重。
- 字体层级必须来自 `MASTER`：标题、正文、说明、表格、数字、按钮、错误提示和移动端文本都应有语义定义。
- 不允许通过任意缩小字号解决字段太多、容器太窄或列宽不足；应调整布局、换行、截断、tooltip、分组、摘要 / 详情或移动端降级。
- 中英文混排、数字、金额、百分比、状态 badge 和按钮文本必须检查基线、行高和垂直居中。
- 圆角、阴影、边框、分割线和按钮尺寸使用同一设计系统尺度，不在同一页面随机变化。
- 图标用于明确动作和状态；不使用 emoji 当结构性图标，不使用无语义装饰图标替代按钮说明。

## 3.1 间距与安全区

- 页面边距、模块间距、组件 padding、表单行距、表格行高和卡片 gap 必须来自 spacing scale。
- 除非 `MASTER` 明确规定，不使用随机 7px、13px、19px 这类无法复用的间距。
- 移动端默认侧向安全边距为 16px 或 `MASTER` 指定值；内容不得贴边。
- 固定 header、footer、bottom action、toast、floating button、drawer 和 modal 必须为内容预留避让空间。
- hover、focus、loading、error 文案、badge 和 tooltip 出现时不得导致布局跳动或遮挡关键内容。

## 4. 状态覆盖

每个关键页面至少验证：

- default：真实或代表性数据可展示。
- loading：骨架、spinner 或占位不会造成布局跳动。
- empty：说明为什么为空，并给出下一步动作或不需要动作的原因。
- error：说明失败原因、恢复动作和重试方式，不泄露内部异常。
- permission：无权限时说明可见性边界和申请 / 返回路径。
- saving / submitting：重复提交被阻止，完成或失败有反馈。

## 5. 表单与输入

- 必填、格式、长度、范围、互斥条件和跨字段校验可见。
- 错误定位到字段或分组，提交失败后用户不丢失已填内容。
- 危险操作有确认和后果说明；批量操作显示影响范围。
- 键盘操作、焦点顺序和移动端输入类型正确。

## 6. 表格、列表与详情

- 表格有稳定列宽、排序状态、筛选回显、分页或虚拟滚动策略。
- 数字、金额、百分比、时间、状态和枚举显示格式一致。
- 行操作、批量操作、详情入口和空数据状态明确。
- 列过多时优先列配置、固定关键列、横向滚动容器或移动端摘要，不让页面整体横向溢出。
- 列表、表格、卡片、详情、记录流和数据看板必须先标明信息优先级：主识别信息、状态、关键指标、次要属性和操作。
- 同类数据必须使用同一模板、同一对齐、同一分隔线、同一行高 / 卡片高度策略和同一数字 / 时间 / 枚举格式。
- 字段过多时使用分组、摘要、详情页、展开区、tooltip、列配置或二级页面；不得单纯压缩字体和间距。
- 移动端表格必须使用卡片、摘要列表、分组详情或局部横向滚动容器；不得产生 body / page 级横向滚动。
- 数据密集页面要匹配产品类型：后台 / 操作台优先扫描效率和稳定布局，landing page 仅在 PRD 明确时采用低密度展示。

## 7. 图表与数据看板

- 图表有标题、时间范围、单位、图例、空态、错误态和数据更新时间。
- 颜色数量可区分，异常值、缺失值和采样口径有说明。
- 图表类型匹配任务：趋势看折线 / 面积，对比看柱状 / 条形，构成看堆叠或 donut，明细仍用表格。

## 8. 导航与流程

- 页面入口、返回、面包屑、tab、深链和权限跳转可解释。
- 当前页面位置明显，局部 tab 不伪装成全局导航。
- 多步骤流程有进度、保存、取消、错误恢复和离开确认。

## 9. 可访问性

- 交互元素可键盘访问，focus 可见。
- 表单控件有 label，图标按钮有 aria-label 或可见文本。
- 对比度满足常规阅读；动效不影响理解，并支持 reduced motion。

## 10. 独立 UI Review

`$pd-ui-review` 生成 `ui/ui-review-report.md` 时，至少检查：

- 产品与设计系统一致性：`MASTER` / Page Overrides、产品范围、页面模式和信息密度。
- 基础视觉质量：风格、字体、颜色、圆角、阴影、边框、图标、对齐和间距。
- 遮挡 / 重叠 / 溢出：固定区、浮层、文本、表格、按钮和移动端横向滚动。
- 交互状态与可访问性：hover、active、focus、disabled、loading、success、error、label、tooltip 和对比度。
- 视口证据：优先记录 375px、768px、1024px、1440px；没有视觉证据时只写证据缺口。
- 同类问题扫描：用户指出或审查发现的问题，要检查同页面、同组件族、同断点和相似 frame / 页面。
- 不能交付条件：没有截图证据、关键 frame 未检查、仍有遮挡 / 重叠 / 溢出、设计系统冲突未说明、P0/P1 未解决时，不得建议进入下一阶段。

## 11. 验收记录

前端完成时，`tech/frontend/frontend-acceptance.md` 至少记录：

- 桌面端视口和移动端视口验证结果。
- 无横向溢出、无文本重叠、关键按钮 / 表单状态完整。
- loading / empty / error / permission / saving 状态验证方式。
- console error、network error、截图或浏览器验证证据。

## 12. Figma 交付前截图 QA

`$pd-figma` 创建或修改 Figma frame 后，交付前必须记录：

- 已检查的 frame / page / component family。
- 截图证据和断点：桌面、平板、移动端；必要时包含 default / loading / empty / error / permission。
- 零容忍问题检查结果：遮挡、重叠、溢出、字体、间距、安全区、状态和产品类型适配。
- 同类问题扫描范围：同页面、同组件族、同断点、相似表格 / 列表 / 卡片 / 表单。
- 待人工确认：无法截图、无法访问 Figma、设计系统缺口或需要产品决策的项。
