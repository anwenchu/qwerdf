# Universal Code Quality Review

用于任意代码级 diff 审查。先确认变更意图，再逐文件看真实行为风险；不要把 formatter、import 顺序或简单 lint 当作主要 finding。

## 正确性

- 检查空值、空数组、缺省字段、边界时间、分页边界、状态转换和分支遗漏。
- 检查 off-by-one、重复提交、幂等、重试、取消、异常路径和回滚路径。
- 检查新增逻辑是否改变已有主流程；对照 PRD、验收标准、技术方案和现有调用方。

## 数据一致性

- 检查事务边界、部分成功、并发写、重复执行、快照字段、唯一约束和业务校验是否一致。
- 写入链路必须明确失败后状态；涉及多资源更新时，检查补偿、回滚或可重试策略。
- 状态枚举、软删除、审计字段和派生字段必须与 API 契约和数据模型一致。

## 错误处理

- 错误码、错误结构和权限失败结果必须符合 `tech/api-contract.md`。
- 不吞异常，不把未知错误误报为成功；后端不泄露内部异常，前端能展示 error / empty / permission 状态。
- 日志应能定位问题，但不得输出凭证、token、敏感信息或完整内部对象。

## 可维护性

- 关注重复逻辑、过长函数、参数膨胀、stringly-typed、隐式耦合、泄漏抽象、难测试分支和无意义封装。
- 新增 helper / service / component 前做复用检查：搜索相邻模块、同层目录和 shared 目录，确认没有已有能力可直接复用或轻量扩展。
- 不要求为偏好重写代码；只有当结构问题带来真实风险、维护成本或测试困难时才升级为 finding。

## 测试质量

- 测试不能只覆盖 happy path；检查失败路径、权限路径、边界数据、并发 / 重试 / 回滚路径。
- 断言必须验证业务结果和副作用，不只验证函数被调用或组件存在。
- 如果 lint / typecheck / test 未执行或失败，把它作为证据缺口或阻断项，而不是代码 finding 的替代品。

## 人工审查边界

- 人工审查优先看业务语义、契约偏差、边界条件、权限、安全、并发、数据一致性、可维护性和测试有效性。
- formatter、import 排序、简单命名偏好、可自动修复 lint 默认不作为 P0/P1；除非能证明它导致行为风险。
- 证据不足时输出“待确认”或残余风险，不把猜测升级为阻断。
