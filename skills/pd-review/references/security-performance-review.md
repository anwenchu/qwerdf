# Security & Performance Review

当 diff 涉及权限、输入、外部调用、日志、敏感字段、数据库写入、网络请求、缓存、队列、批处理或性能敏感路径时读取。

## 安全与隐私

- 检查越权和 IDOR：用户传入的 accountId、userId、orderId、tenantId、resourceId 必须与当前身份和权限域绑定。
- 输入校验必须覆盖类型、长度、范围、枚举、文件类型、URL、分页参数和排序字段；不要只依赖前端校验。
- 注入风险包括 SQL / NoSQL / shell / template / log injection；动态查询、排序字段和 raw SQL 必须有白名单或参数化。
- 前端 XSS 风险：HTML 注入、富文本、URL 参数、第三方脚本、dangerouslySetInnerHTML、未转义 markdown。
- SSRF / 外部调用风险：用户可控 URL、重定向、内网地址、metadata IP、超时、重试和响应大小限制。
- 敏感字段回显和日志泄密：token、密码、密钥、证件、手机号、邮箱、地址、内部配置和完整请求 / 响应体默认不输出。
- 凭证处理必须使用环境变量或安全存储；不得提交 `.env`、密钥、私有证书或 hardcoded token。

## 性能

- 后端重点看 N+1 查询、无界查询、缺失分页、全表扫描、重复远程调用、阻塞式 IO、锁粒度过大和批处理内存膨胀。
- 前端重点看重复请求、数据 waterfalls、过度重渲染、大 bundle、同步重计算、未清理订阅 / timer 和缓存失效。
- 缓存必须有失效策略、权限隔离和错误降级；不要缓存用户敏感数据到跨用户可见区域。
- 分页和排序必须稳定；无限滚动、导出、批量操作和列表筛选要有上限和取消策略。

## 严重性参考

- 可被未授权用户读取 / 修改他人数据、执行高危操作或泄露敏感字段，通常为 `P0` / `P1`。
- 主流程写入出现部分成功、重复扣减、重复创建、事务不一致或无法恢复迁移，通常为 `P0` / `P1`。
- 性能问题如果会使核心接口或首屏在正常数据量下不可用，可升为 `P1`；仅在规模增长后可能退化，通常为 `P2`。
- 证据不足时标记待确认，说明需要的日志、压测、SQL explain、浏览器 profile 或安全验证。
